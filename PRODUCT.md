# PulseOps — Incident Intelligence Console

## Идея

PulseOps — демонстрационная панель для наблюдения за состоянием SaaS-продукта и координации инцидентов. Пользователь видит сервисы и метрики в реальном времени, замечает деградацию, открывает инцидент, изучает временную шкалу и назначает ответственных.

Это не статичный лендинг: проект должен ощущаться как небольшой законченный продукт, которым действительно можно пользоваться.

## Почему проект подходит для портфолио

- Показывает сложное клиентское состояние, а не только вёрстку.
- Даёт Redux реальную роль: события, фильтры, уведомления, выбранный инцидент и настройки интерфейса.
- Демонстрирует строгую типизацию домена и асинхронных сценариев.
- Включает графики, таблицы, поиск, фильтры, optimistic UI и симуляцию live-данных.
- Позволяет показать производительность, доступность, адаптивность и тестирование.
- Имеет запоминающуюся визуальную тему, но остаётся правдоподобным рабочим интерфейсом.

## Главный сценарий демо

1. Пользователь открывает обзор и видит, что все сервисы работают штатно.
2. Встроенная симуляция создаёт всплеск latency и рост error rate у Payments API.
3. Система поднимает инцидент и показывает уведомление.
4. Пользователь проваливается в детали, фильтрует события и изучает временную шкалу.
5. Пользователь подтверждает инцидент, назначает владельца и добавляет заметку.
6. После стабилизации метрик инцидент переводится в resolved, а интерфейс показывает итоговые показатели MTTA/MTTR.

## Экраны MVP

### 1. Overview

- общий статус системы;
- карточки latency, error rate, throughput и active incidents;
- график за выбранный временной диапазон;
- список сервисов с health/status;
- лента последних событий;
- запуск демонстрационного сбоя.

### 2. Incidents

- список инцидентов;
- поиск, фильтрация по severity/status/service;
- сортировка и сохранённые представления;
- понятные empty/loading/error states.

### 3. Incident details

- заголовок, severity, status и owner;
- связанный график метрик;
- временная шкала событий;
- заметки и смена статуса;
- краткое резюме причины и результата.

### 4. Services

- каталог сервисов;
- зависимости между сервисами;
- SLA/SLO и последние деплои;
- drill-down до истории выбранного сервиса.

## Визуальное направление

Стиль: тёмный editorial/operations UI с тёплым почти белым текстом, графитовой поверхностью и одним сигнальным лаймовым акцентом. Severity использует отдельные функциональные цвета. Основа — строгая сетка, заметная типографика и плотный рабочий интерфейс.

React Bits используем дозированно:

- `Grid Scan` или `Dot Grid` — очень слабый фон hero/empty state;
- `Decrypted Text` — короткое появление системного статуса;
- `Border Glow` — для ключевых KPI и активного инцидента; свечение появляется при hover/focus и меняет цвет в зависимости от severity;
- `Dock` — мобильная навигация;
- `Scroll Reveal` — для презентационной страницы проекта.

Не подсвечиваем все карточки постоянно: обычные поверхности сохраняют тонкую статичную рамку, а `Border Glow` обозначает интерактивность или важное состояние. Не используем одновременно несколько активных shader/background эффектов. Все анимации учитывают `prefers-reduced-motion`, а данные и контраст остаются важнее декора.

## Стек

### Основа

- React 19;
- TypeScript в strict mode;
- Vite;
- React Router;
- CSS Modules + CSS custom properties;
- ESLint + Prettier.

### Состояние и данные

- Redux Toolkit;
- RTK Query для запросов и кеша;
- listener middleware для сценария симуляции инцидента;
- MSW как реалистичный mock API на первом этапе;
- Zod для проверки данных на границе API.

### Backend

Backend появится после готового frontend MVP и останется небольшим:

- Cloudflare Pages Functions + Hono + TypeScript;
- Cloudflare D1 (SQLite);
- Drizzle ORM и несколько миграций;
- REST API для сервисов, инцидентов, событий и заметок;
- без авторизации в MVP;
- live-метрики симулируются на клиенте, поэтому WebSocket/SSE-сервер не нужен.

Frontend и API разворачиваются как один проект на Cloudflare. Для портфолио этого достаточно: есть настоящая база и CRUD, но нет отдельного сервера, Docker и постоянно оплачиваемой инфраструктуры. Сначала RTK Query работает с MSW по тому же контракту, который затем реализует API.

### UI и визуализация

- React Bits — отдельные анимированные примитивы, копируемые в код проекта;
- Recharts для временных рядов;
- Lucide React для иконок;
- Radix UI только для доступных сложных примитивов: dialog, tooltip, select;
- date-fns для времени и диапазонов.

### Качество

- Vitest + React Testing Library;
- Playwright для основного пользовательского сценария;
- Storybook — после MVP, для ключевых состояний компонентов;
- axe для базовых accessibility-проверок.

## Архитектура

Используем feature-oriented структуру без преждевременной сложности:

```text
src/
  app/          # store, router, providers
  pages/        # композиция экранов
  features/     # incident filters, simulation, time range
  entities/     # incident, service, metric, event
  shared/       # UI, utils, API, tokens
```

Серверные данные хранятся в RTK Query. Redux slices используются только для состояния интерфейса и сценариев, которым действительно нужен общий источник истины. Локальное состояние формы остаётся внутри компонентов.

## Этапы разработки

1. **Foundation** — Vite-проект, строгий TS, токены, шрифты, базовые компоненты и AppShell.
2. **State architecture** — store, typed hooks, RTK Query, MSW и доменные типы.
3. **Overview** — KPI, service health, chart, activity feed и responsive layout.
4. **Live simulation** — управляемый поток событий, деградация и восстановление сервиса.
5. **Incident workflow** — список, фильтры, details, timeline и смена статуса.
6. **Services** — каталог и визуализация зависимостей.
7. **Backend** — Pages Functions, Hono, D1, миграции и замена MSW на реальный transport.
8. **Polish** — React Bits, micro-interactions, reduced motion, keyboard navigation.
9. **Verification** — unit/component/E2E тесты, Lighthouse и финальный README с GIF/скриншотами.

## Definition of done для MVP

- основной сценарий полностью работает без настоящего backend;
- интерфейс адаптивен от 360 px до desktop;
- нет ошибок TypeScript, lint и тестов;
- вся интерактивность доступна с клавиатуры;
- анимации отключаются системной настройкой;
- проект разворачивается одной командой;
- README объясняет архитектурные решения, а не только установку.
